1. Transformer 核心架构

落地问题: "为什么这个模型快/慢? 为什么长文本掉精度? 为什么KV Cache是瓶颈?"
  • Self-Attention 公式: Q·Kᵀ/√dₖ → Softmax → ×V
  • Multi-Head Attention 的物理含义 (每个头看不同子空间)
  • Positional Encoding: 绝对位置 vs RoPE (旋转位置编码)
  • 残差连接 + LayerNorm (Pre-Norm vs Post-Norm)
  • FFN (前馈网络): 为什么占参数量 2/3
  • ⚠ 避坑: 不需要手写反向传播, 但必须能手绘数据流

2. 显存经济学 (GPU Memory Budget)

落地问题: "为什么70B模型跑不动? 量化到4bit省多少显存? 换A100还是H20?"
  • 显存构成 = 模型参数 + 优化器状态 + 中间激活 + KV Cache
  • 模型参数: FP32 = 4B×参数量, FP16 = 2B×, INT4 = 0.5B×
  • 优化器状态: Adam = 参数量×12B (moments+variance+param)
  • KV Cache 计算公式: 2×层数×头数×头维度×序列长度×batch×dtype
  • 推理 vs 训练显存对比 (训练需要反向传播, 约推理的3-4倍)
  • ⚠ 务必实测: nvidia-smi + PyTorch memory_allocated()

3. 推理 (Inference) vs 训练 (Training)

落地问题: "客户要的是推理服务, 为什么还要懂训练? 预填充和自回归有什么区别?"
  • 推理两阶段: Prefill (预填充, compute-bound) + Decode (自回归, memory-bound)
  • 训练: 前向+反向+梯度更新, 需要完整计算图
  • 为什么推理瓶颈是显存带宽, 不是算力
  • TTFT (首Token时间) vs TPOT (每Token时间) 的影响因素
  • Continuous Batching (连续批处理): vLLM的实现原理
  • 推测解码 (Speculative Decoding): 用小模型猜, 大模型验证

4. Token 经济学

落地问题: "为什么Token这么贵? 怎么降? Context Window越大成本怎么增长?"
  • Token ≠ 字: 中文约1.5-2字/Token, 英文约0.75词/Token
  • Attention复杂度: O(n²) — 序列翻倍, 计算量翻4倍
  • 输入Token (prompt) vs 输出Token (completion) 的成本差异
  • KV Cache 随序列长度线性增长 → 长对话终极瓶颈
  • Token压缩策略: Prompt压缩、上下文摘要、滑动窗口
  • 成本估算公式: (输入Token×单价入 + 输出Token×单价出) × 并发
  • ⚠ 实战: 用 tiktoken/Tokenizer 实测各模型Token计数

5. PyTorch 工程基础

落地问题: "为什么模型加载就OOM? 怎么把模型放多张卡? 怎么debug一个推理bug?"
  • Tensor 基础: 创建、reshape、device移动、dtype转换
  • CUDA 显存管理: .to("cuda") / .cpu() / torch.cuda.empty_cache()
  • 模型加载: state_dict vs 完整模型, HuggingFace AutoModel
  • 推理模式: model.eval() + torch.no_grad() (必须, 否则显存翻倍)
  • 数据管道: DataLoader, collate_fn, pin_memory
  • ⚠ 够用即可: 不需要写自定义Layer/自定义Loss

6. 推理引擎生态

落地问题: "vLLM和TGI怎么选? Ollama能用吗? 什么时候上TensorRT-LLM?"
  • vLLM: PagedAttention + Continuous Batching, 吞吐量王者
  • TGI (Text Generation Inference): HuggingFace官方, 生态好
  • Ollama: 本地部署首选, 但生产环境谨慎
  • llama.cpp: CPU推理 + GGUF量化, 边缘设备首选
  • TensorRT-LLM: NVIDIA官方, 极致优化但工程复杂度高
  • 选型矩阵: 吞吐量 vs 延迟 vs 工程成本 vs 硬件兼容

7. 模型量化 (Quantization)

落地问题: "GPTQ和AWQ选哪个? INT8还是INT4? 量化后效果掉多少?"
  • 原理: 把FP16权重映射到低精度 (INT8/INT4/NF4)
  • 对称量化 vs 非对称量化
  • GPTQ: 逐层量化+权重重排, 适合GPU, 需要校准数据
  • AWQ: 激活感知, 保护关键权重通道, 精度损失更小
  • GGUF/GGML: llama.cpp的量化格式, Q4_K_M 是甜点
  • Bitsandbytes: NF4 + 双量化, QLoRA训练标配
  • ⚠ 核心认知: 量化的本质是用精度换显存, 但不同层的敏感度不同

8. Fine-tuning 方法论

落地问题: "全量微调还是LoRA? 什么时候用QLoRA? 数据要多少条?"
  • 全量微调(Full FT): 更新所有参数, 效果好但贵 (70B需~560GB显存)
  • LoRA: 在旁路加低秩矩阵, 只训练~1%参数, 效果接近全量
  • QLoRA: LoRA + 4bit量化基础模型, 单卡24GB可训70B
  • 数据量经验法则: 1K条起步, 5K-10K收敛, 超过50K看收益递减
  • 指令微调(Instruction FT) vs 偏好对齐(DPO/RLHF)
  • 灾难性遗忘: 混合通用数据防退化
  • ⚠ 实战优先级: QLoRA先跑通 → 效果好就够 → 不够再上全量

9. Embedding 体系

落地问题: "RAG为什么检索不准? 怎么选Embedding模型? 多语言怎么处理?"
  • Embedding = 把文本映射到高维向量 (768/1024/4096维)
  • 相似度: Cosine Similarity (最常用) / 欧氏距离 / 点积
  • MTEB榜单: 模型选型依据 (关注Retrieval/Clustering/Classification)
  • BGE系列 (中文强) / text-embedding-3 (OpenAI) / nomic (本地)
  • Chunk策略对检索的影响 >> Embedding模型本身
  • Late Chunking: 先整篇Embedding再切分, 提升长文档检索
  • 多语言: 多语言模型 vs 翻译后检索 vs 双塔模型

10. Attention 机制进阶

落地问题: "为什么长文本慢? FlashAttention省了多少? GQA和MHA有什么区别?"
  • 标准Attention复杂度 O(n²d) — 序列长度的平方
  • FlashAttention: 利用GPU SRAM分块计算, 2-4x加速, 显存O(n)
  • MHA (Multi-Head): 每个头独立QKV, 效果最好, 显存最大
  • MQA (Multi-Query): 所有头共享KV, 省显存, 精度略降
  • GQA (Grouped-Query): 分组共享KV, MHA和MQA的折中 (Llama2/3用)
  • GQA公式: 头数/组数 = 每个KV组服务的Query头数
  • ⚠ 选型: 推理优先→GQA, 训练优先→MHA, 边缘设备→MQA

11. 推理优化技术栈

落地问题: "为什么同样的模型, 别人的API比我便宜一半? 延迟怎么压到200ms?"
  • 算子融合 (Kernel Fusion): 把多个CUDA kernel合并, 减少显存IO
  • PagedAttention: vLLM的核心, 把KV Cache按页管理, 减少碎片
  • Prefix Caching: 共享前缀只算一次 (System Prompt场景收益巨大)
  • Dynamic Batching: 不等batch满就发车, 降低排队延迟
  • FP8推理: H100/H200原生支持, 比FP16快2x
  • 模型并行: 张量并行(TP) + 流水线并行(PP) + 数据并行(DP)
  • ⚠ 够用即可: 知道什么场景用什么技术, 不需要手写CUDA kernel

12. 模型评估与选型

落地问题: "这个模型到底好不好? 怎么跟客户证明? 换了量化版效果变了吗?"
  • 基准测试: MMLU(知识) / HumanEval(代码) / MT-Bench(对话)
  • 但落地不看榜单: 用客户真实数据做A/B测试
  • 评估维度: 准确率 + 延迟 + 成本 + 稳定性
  • Prompt Sensitivity: 同一个模型, 不同Prompt结果差距 > 模型间差距
  • 长尾评估: 需要人工抽检, 指标看不出"偶尔胡说"
  • 量化评估: 同模型INT4 vs FP16 在具体任务上的退化曲线
  • ⚠ 铁律: 用真实业务数据跑, 不只看榜单

13. 知识蒸馏 (Knowledge Distillation)

落地问题: "怎么用大模型教小模型? 客户想省钱但又想要70B的效果怎么办?"
  • Teacher-Student: 大模型(Teacher)输出logits教小模型(Student)
  • 黑盒蒸馏: 只用Teacher的输出文本训练Student (最常用)
  • 白盒蒸馏: 用Teacher的中间层/logits分布 (效果更好, 需要模型权重)
  • 蒸馏vs微调: 蒸馏学的是推理能力, 微调学的是格式服从
  • 合成数据: 用大模型生成训练数据 → 训练小模型 → 循环迭代
  • 典型案例: GPT-4 → LLaMA3 蒸馏流水线

14. MoE (混合专家模型)

落地问题: "Mixtral 8x7B为什么快? DeepSeek-V3为什么便宜? 要全量部署吗?"
  • 核心思想: 把FFN拆成多个"专家", 每次只激活2-3个
  • Router (门控网络): 决定每个Token走哪个专家
  • 稀疏激活: Mixtral 8x7B 总参数46.7B, 每次只激活12.9B
  • 显存: 仍需加载所有专家 → 显存按总参数量算, 计算按激活量算
  • 负载均衡: 专家不能"忙的忙死闲的闲死" (auxiliary loss)
  • 落地取舍: MoE省算力不省显存, 适合云端推理, 不适合边缘

15. RLHF / DPO 对齐训练

落地问题: "模型为什么会有'价值观'? 怎么让模型更符合企业调性? DPO是什么?"
  • RLHF三阶段: SFT → Reward Model训练 → PPO强化学习
  • Reward Model: 用人类偏好数据训练一个打分模型
  • PPO: 用Reward Model的信号, 通过强化学习调整模型
  • DPO (Direct Preference Optimization): 跳过Reward Model, 直接从偏好对学习
  • DPO优势: 更简单、更稳定、不需要单独训练Reward Model
  • 落地: 企业调性对齐 → 收集偏好数据 → DPO训练 → 效果验证
  • ⚠ 够用即可: 理解原理和差异, 实际训练交给算法团队

16. 多模态基础

落地问题: "FLUX的UNet在干什么? CLIP为什么是Vision和Language的桥梁?"
  • CLIP: 用对比学习把图文映射到同一向量空间
  • ViT (Vision Transformer): 把图片切成Patch, 当Token喂给Transformer
  • Diffusion模型: 前向加噪 → 反向去噪 (UNet预测噪声)
  • Stable Diffusion: VAE压缩 → UNet在潜空间去噪 → VAE解码
  • FLUX: 你已经在用了, 补一下UNet/VAE/CLIP三个组件的职责
  • 多模态LLM: 视觉编码器 + 投影层 + LLM (如LLaVA架构)
  • ⚠ 你已踩过的坑就是最好的教材: denoise强度、CFG scale、sampler选择

17. Agent 相关的模型能力

落地问题: "为什么有些模型不适合做Agent? Tool Calling为什么经常失败?"
  • Function Calling: 模型输出结构化JSON的能力 — 不是所有模型都有
  • ReAct / CoT: 推理链 — 模型需要被训练/提示才会"一步步想"
  • 上下文窗口利用: Agent场景Token消耗极大, 模型能否有效利用长上下文
  • 幻觉率: Agent场景幻觉代价高 (错误Tool调用 → 连锁错误)
  • 指令遵循: 越强的模型不一定越听话 (需要对齐训练)
  • 选型基准: 在Agent Benchmark上测试, 不是MMLU

18. 模型安全与护栏

落地问题: "怎么防Prompt注入? 企业客户的第一诉求永远是安全合规。"
  • Prompt注入: 用户通过输入操纵系统提示 → 输入清洗 + 角色分离
  • 越狱(Jailbreak): 绕过安全限制 → 输入层+输出层双层护栏
  • 数据泄露: 模型可能"记住"训练数据 → 数据脱敏 + 输出过滤
  • Guardrails: NeMo Guardrails / Guardrails AI / 自定义规则引擎
  • 落地: 输入过滤 → 模型推理 → 输出审核 三层防护

核心原则重申:
1. 每个知识点必须能回答一个落地问题。学Transformer不是为了发论文, 是为了跟算法说"这个位置编码换成RoPE能不能提升长文本?"
