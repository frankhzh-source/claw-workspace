## 一、整体关系：你站在哪一层

先搞清楚你在这个系统里的位置。整个产品分三层：

```
┌──────────────────────────────────────────────────────────────┐
│  ① 前端工作台（PRD里的部分）                                │
│  运营上传商品 → 选参数 → 提交任务 → 看结果 → 评分 → 审核   │
│  跟你没关系，这是前端+后端团队做的                          │
└──────────────────────┬───────────────────────────────────────┘
                       ↓ 提交生成任务
┌──────────────────────────────────────────────────────────────┐
│  ② 模型通道适配器（PRD里的 D3 组件）                       │
│  统一接口，屏蔽不同模型的差异                                │
│  云端API → 中转站 → 你的本地模型 → 未来的视频模型           │
│  你只需要让你的模型符合这个接口规范                          │
└──────────────────────┬───────────────────────────────────────┘
                       ↓ 发送推理请求
┌──────────────────────────────────────────────────────────────┐
│  ③ 你的本地模型（**你负责的部分**）                         │
│                                                              │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐               │
│  │ LoRA训练  │    │ 模型管理  │    │ 推理服务  │               │
│  │ 你训练    │ →  │ 存+版本化 │ →  │ HTTP接口  │               │
│  │ LoRA模型  │    │ 管理权 重 │    │ 对外提供  │               │
│  └──────────┘    └──────────┘    └──────────┘               │
└──────────────────────────────────────────────────────────────┘
```

**你的核心职责只有第三层：**
- **训练 LoRA**（训练多个 LoRA，每个对应一个人物+商品组合）
- **管理 LoRA**（把训练好的权重文件存好、打标签、版本化）
- **推理服务**（启动一个 HTTP 服务，接收平台的请求，用对应 LoRA 生图，返回结果）

**你不需要做的事情：**
- ❌ 不做前端页面
- ❌ 不做任务队列
- ❌ 不做审核工作台
- ❌ 不做数据库

**一句话定位你的角色：**
> 你是**模型通道**的提供方。平台把"任务参数"丢给你，你把"图片"还回去。中间用哪个 LoRA、调哪些参数，是你的事。

## 二、PRD 里的任务类型 vs 你的三层生图方案

PRD 定义了 4 类任务。你的三层方案正好对应其中 3 类：

| PRD 任务类型 | 你的三层对应的哪层 | 说明 |
|-------------|-------------------|------|
| **商品主图** | L3 多场景→主图 | 白底图+场景，人穿衣服站姿/坐姿 |
| **详情页/场景图** | L3 多场景→场景 | 多角度、特写、环境融入 |
| **细节图** | L3 多场景→细节 | 面料、纹理、领口、袖口 |
| **上身/三视图** | **L2 人物三视图** | 人穿衣服的正/侧/背三视角 |

**但有一个关键差异：PRD 的用户流程和你想象的顺序是反的。**

```
PRD 用户视角（运营人员的操作顺序）：
上传商品图 → 选「上身/三视图」任务 → 出图 → 选「主图」任务 → 出图

你的训练视角（模型训练的顺序，必须先做完前一步才能做后一步）：
训练 L1（正反图）→ 训练 L2（三视图，按人物）→ 推理 L3（多场景，复用已有 LoRA）
```

**这意味着：** 你的 LoRA 训练周期是**异步的**——运营在前台点任务，你在后台跑训练。训练好的 LoRA 要提前准备好，不能等运营点了才开练。

## 三、完整工作流程（你实际要做的事）

### 阶段 A：数据准备（你独立完成）

```
商品入库（运营上传到平台）
    ↓
你定期从平台拉取新商品的图片 + 商品信息
    ↓
L1 - 训练一个「商品结构感知」LoRA
    - 输入：商品正反面白底图 × 3-5 张
    - 训练目标：学会这个商品的外形轮廓
    - 输出：LoRA 权重文件 A.safetensors
    ↓
L2 - 选3-5个模特，每人训练一个「人物特征」LoRA
    - 输入：每个人物的正/侧/背面照片 × 10 张
    - 训练目标：学会这个人的体态特征
    - 输出：LoRA 权重文件 B1.safetensors, B2.safetensors, ...
```

### 阶段 B：推理服务（自动运行）

```
平台提交了一个任务：
  "商品ID: 黑色LOVE睡衣，任务类型: 上身/三视图，模特: 年轻女性，风格: 甜美"

你的推理服务收到请求后：
    ↓
Step 1 - 加载 LoRA
    Load(黑色LOVE睡衣_结构感知.safetensors)
    Load(年轻女性_人物特征.safetensors)
    ↓
Step 2 - 组装 Prompt（模板写好，直接套用）
    "年轻女性穿着黑色LOVE字母睡衣套装，正面站姿，商业摄影..."
    ↓
Step 3 - 推理生成
    FLUX + 合并的 LoRA → 生成 3 张（正/侧/背）
    ↓
Step 4 - 返回图片 URL 给平台
    { status: "success", images: ["url1", "url2", "url3"], cost: 0.05 }
```

### 阶段 C：迭代优化（持续进行）

```
运营+设计在平台上评分审核
    ↓
你会看到各 LoRA 的「通过率」「平均分」
    ↓
低于阈值的 LoRA 需要重新训练（加数据、调参数）
    ↓
新版本保存为 V2，平台自动用新版本生成
```

## 五、三层生图方案在推理时的详细逻辑

这个是最关键的——你训练好 LoRA 之后，**怎么用它们出图**。

### 场景1：L1+L2 → 上身/三视图

```
平台请求：上身/三视图，模特A穿黑色LOVE睡衣

你的推理服务做：
1. 加载 base model（FLUX.1-dev）
2. 合并 LoRA：商品结构感知_L1 + 模特A人物特征_L2
    → lora_scale 各 0.7-0.8（防止过拟合）
3. 用模板 A 生成：
     [模特描述, 服装描述, 姿态:正面/侧面/背面, 灯光:商业棚拍]
4. 出 3 张图（正/侧/背）
5. 返回
```

### 场景2：L1+L2+L3 → 详情页/场景图

```
平台请求：场景图，模特A在卧室穿黑色LOVE睡衣

推理逻辑和上面一样，但：
1. 加载的 LoRA 不变（商品结构 + 人物特征就够了）
2. Prompt 换场景模板：
     "模特A, 穿着黑色LOVE睡衣, 坐在卧室床边, 窗边自然光..."
3. 所以 L3 不需要单独训练 LoRA！
   它只是在 L1+L2 的基础上换 Prompt 而已
```

**核心理解（这一点最重要）：**

| 你想要的效果 | 需要训练的 LoRA | 推理时改什么 |
|-------------|----------------|------------|
| 商品穿在A身上（上身图） | L1商品结构 + L2人物特征A | 改姿态词 |
| 商品穿在A身上（正面站姿） | 同上 | 改姿态词 |
| A在卧室场景 | 同上 | 改场景词 |
| A在室外场景 | 同上 | 改场景词 |
| A穿不同颜色变体 | 同上 | 改颜色词 + 参考图 |
| B（另一个模特）穿同一件 | L1 + L2人物特征B | 换 L2 LoRA |

**L3「多场景扩展」并不需要额外训练 LoRA！** 它只是用**已经训练好的 L1+L2 组合**，配合不同的 Prompt 来出图。只有在场景特别特殊（比如某些光线条件 LoRA 学不会）时，才需要专门训练一个「场景辅助 LoRA」。

```
【循环开始】

① 运营上传了新商品 → 你拿到货了
     ↓

     → 没有 → 训练 L2 人物特征 LoRA（1-2小时）
     → 已有 → 跳过
     ↓

     → 你什么都不用做，推理服务自动用最新版本
     ↓
⑤ 运营+设计评分审核
     → 通过率低 → 需要优化 → 回到②或③
     → 通过率高 → 不做任何事
     ↓
⑥ 每隔一段时间，检查所有 LoRA 的使用数据
     → 长期不用的 → 冷存储
     → 效果下降的 → 重新训练

【循环结束】
```

## 十、你需要理解的几个关键概念

> 以下是一个全栈开发视角的解读，帮你理解这个系统怎么运作。

### 概念1：LoRA 本质上是什么？
你训练出来的 `.safetensors` 文件 = **一个补丁包**。它不包含整个模型，只包含"把原模型往某个方向微调"的差异信息。所以：
- 叠加商品 LoRA → 模型更懂这件衣服的结构
- 叠加人物 LoRA → 模型更懂这个模特的长相
- 两个可以同时叠加 → 模型同时懂衣服和模特

### 概念2：模型通道适配器解决了什么问题？
如果没有这个适配器，每个模型接入都要给平台改代码。有了适配器：
- 平台不关心你用 FLUX 还是 Midjourney 还是本地模型
- 它只发固定格式的请求，收固定格式的回复
- 你换模型、加模型、升版本，都只改你的推理服务代码，平台不用动

### 概念3：为什么 L3 不需要单独训练？
L1 让模型学会"这个商品的外形"，L2 让模型学会"这个模特的特征"。L3 只是**改变 Prompt 的描述词**（"室内"→"室外"，"站姿"→"坐姿"），不需要额外的 LoRA。只有在极少数情况下（比如某种光线条件 FLUX 的基础能力不够），才需要训练一个 L3 辅助 LoRA。

### 概念4：你的 GPU 一次只能处理一个任务
24GB 显存跑 FLUX 推理+LoRA，一次大概占 12-16GB。如果你同时处理两个任务，显存可能爆。所以需要一个简单的队列——任务一个个来，排队的等着。
- 平台把任务发给你
- 你回 `{status: "processing"}`
- 处理完再回调通知平台
- 平台那边展示"生成中"，用户不会卡住
