# 飞书文档创建策略 · 比较与选择

## 7 种可行策略全景

| 策略 | 别名 | 一句话描述 |
|------|------|-----------|
| A | 全量 append | `create <title>` → 一次 `--command append` 写完 |
| B | 全量 create | `create --content` 一次性传完整内容 |
| C | 骨架逐块 insert | 先创建骨架 heading → 逐个 `block_insert_after` |
| D | 分批 append | `create <title>` → 多次 `--command append` 分批追加 |
| E | MCP 导入 | `mcp__feishu-docx__docx_builtin_import` 直接导入 markdown |
| F | 原生 OpenAPI | 直接调飞书 CreateDocument + AddBlock |
| G | 本地上传 + 转换 | 本机生成 MD，经 lark-drive 上传为飞书文档 |

## 各策略详解

### 策略A：全量 append（当前推荐）
```
lark-cli docs +create --content '<title>文档标题</title>'
lark-cli docs +update --command append --content '完整XML内容...'
```
**特点：**
- 创建时只写标题（零骨架风险）
- 全量内容一次性 append 完成
- 不涉及 block 树结构调整
- 后续追加只能 append 到末尾

**局限：**
- 内容过长时可能会触发 CLI 参数长度限制（实测 50,000+ 字符 OK）
- 无法在文档中间插入内容

---

### 策略B：全量 create（最直白）
```
lark-cli docs +create --content '<title>文档标题</title>\n<h1>完整内容...</h1>'
```
**特点：**
- 一步到位，创建即完成
- 代码最短，心智负担最小
- 无需二次命令

**局限：**
- 创建时出错需要重新创建（无恢复）
- XML 内容含特殊字符时容易解析失败
- 之前遇到过 400 错误（XML 格式问题而非长度）

---

### 策略C：骨架逐块 insert（踩坑方案）
```
lark-cli docs +create --content '<title>...</title><h1>...</h1><h1>...</h1>...'
lark-cli docs +update --command block_insert_after --block-id xxx --content '...'
lark-cli docs +update --command block_insert_after --block-id yyy --content '...'
```
**特点：**
- 精确控制每个块的位置
- 可以插入到任意已有块之后

**致命问题：**
- `block_insert_after` 在目标 heading 已有子节点时，会覆盖部分兄弟节点
- 是本次反复空白/错位的根因
- **建议永久弃用**

---

### 策略D：分批 append（灵活版A）
```
lark-cli docs +create --content '<title>文档标题</title>'
lark-cli docs +update --command append --content '第一批内容...'
lark-cli docs +update --command append --content '第二批内容...'
lark-cli docs +update --command append --content '第三批内容...'
```
**特点：**
- 比策略A更灵活，可分多批次写完
- 每批次都是 append 到末尾，安全
- 适合内容分段生成（如 AI 分批产出）

**局限：**
- 批次间顺序依赖，不能后补中间内容
- 分批次增加了操作次数和时间

---

### 策略E：MCP 导入（最简洁）
```python
# 通过 MCP 工具直接导入 markdown
mcp__feishu-docx__docx_builtin_import(content=md_content)
```
**特点：**
- 直接传 markdown，不写 XML
- 一步完成创建+内容填充
- 对 markdown 格式支持良好

**局限：**
- 之前遇到过 400 错误（特定字符兼容问题）
- 对大文档可能超限
- 没有中间步骤做校验

---

### 策略F：原生 OpenAPI（最底层）
```
POST https://open.feishu.cn/open-apis/docx/v1/documents
POST https://open.feishu.cn/open-apis/docx/v1/documents/:id/blocks
```
**特点：**
- 最底层的控制力
- 增量更新、精确位置插入都支持
- 可编程控制每个 block 的属性

**局限：**
- 需要自行管理 token 和权限
- 学习成本高（block 树结构复杂）
- 需要写代码而非 CLI 命令

---

### 策略G：本地上传 + 转换
```
# 本地写好 markdown
# 上传到飞书云空间
lark-cli drive upload --path 本地文件 --destination 飞书目录
# 或手动上传后转换为飞书文档
```
**特点：**
- 可离线编辑，确认无误后再上传
- 上传即备份
- 不依赖飞书文档 API 的稳定性

**局限：**
- 多一步上传操作
- 格式转换可能有损失（本地 MD → 飞书文档）
- 无法精确控制排版

---

## 决策树：选哪个策略？

```
创建场景是什么？
│
├─ 一次性写入，不需要后续修改
│  ├─ 内容 ≤ 50000 字符 → 策略B（全量 create）
│  └─ 内容 > 50000 字符 → 策略A（全量 append）
│
├─ 需要分批追加（AI 逐步产出）
│  └─ 策略D（分批 append）
│
├─ 已有 markdown 文件
│  ├─ 格式简单 → 策略E（MCP导入）
│  └─ 格式复杂 → 策略A（全量 append，转 XML 写入）
│
├─ 需要精确控制每个块位置
│  ├─ 短期项目 → 策略F（原生API）
│  └─ 长期项目 → 策略D + 策略F 混合
│
├─ 需要离线先编辑确认
│  └─ 策略G（本地上传）
│
└─ 内容 ＜ 3000 字、一次性
   └─ 策略A 或 策略B
```

## 按场景推荐

| 你的场景 | 推荐策略 | 理由 |
|---------|---------|------|
| 写公众号文章正文 | A. 全量append | 安全，内容完整，不涉及后改 |
| 写连载/持续更新的资料 | D. 分批append | 每次追加到末尾，不干扰前面 |
| 快速测试/量少内容 | B. 全量create | 一步到位，最快 |
| 本地写好再上传 | G. 本地上传 | 离线确认，无 API 风险 |
| 编程写入大量结构化内容 | F. 原生API | 最底层的控制力 |
| 已有的 markdown 文件 | E. MCP导入 | 最简洁的路径 |
| **需要插入到文档中间** | ⚠️ **C 弃用，改用 D+F** | 必须用原生API |

---

## 最优策略总结

| 优先级 | 策略 | 适用场景 |
|:-----:|------|---------|
| 🥇 | A. 全量append | **通用首选**，大部分场景的最佳平衡 |
| 🥇 | B. 全量create | 内容少、一次性、不追回 |
| 🥈 | D. 分批append | AI 持续产出、连载更新 |
| 🥉 | E. MCP导入 | 已有 markdown 文件 |
| ❌ | C. 骨架insert | **永久弃用** - block_insert_after 有结构性 BUG |
