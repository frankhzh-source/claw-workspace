OpenClaw 执行问题 · 四层解法框架
核心根源：AI 模型在"离线推理"，缺乏执行时的实时感知与反馈
问题一：长周期任务执行遗漏
Agent 是无状态单次推理，没有持续运行进程、没有真正记忆、没有时钟驱动。定时任务 = 触发一次对话，不是真正的工作流引擎。
问题二：可视化图反复调整
SVG 生成是纯文本推理，模型不知道文字实际渲染多宽多高，无视觉反馈闭环，只能盲目猜测坐标，靠用户反馈校正。
解法四层框架
第一层 工程补偿 · 现在就能做
  • 长任务分段 + 检查点：每段结束写入文件，下次从文件读取状态继续
  • 任务结果验证步骤：自动化任务结束后主动检查是否真正完成
  • 可视化改用 HTML 输出：CSS Flexbox/Grid 让浏览器处理布局，比 SVG 坐标稳定
第二层 模型升级 · 有帮助但非根本
  • 更强模型在 SVG 空间计算上更好，但"不知道渲染结果"的物理限制不消失
  • 上下文窗口越大，长任务遗漏概率降低，但不等于零
  • 多模态模型（截图回传自我修正）是真正有帮助的方向，OpenClaw 尚未接入
第三层 架构根本解 · 需平台支持
针对长任务遗漏
  • 持久化任务状态：进度存数据库，Agent 每次启动先读状态
  • 任务编排引擎：类 Airflow/Temporal，有 DAG、重试、失败回滚
针对可视化布局
  • 浏览器渲染反馈闭环：生成 HTML → headless browser 截图 → 传回模型修正 → 循环
  • OpenClaw 有 Agent Browser Skill，理论可行，需串入生成流程
第四层 行业趋势 · 2–3 年方向
  • Computer Use：模型直接操作屏幕，"看着屏幕操作"，布局问题从根本消失（Anthropic 在做）
  • 持久 Agent：真正 24 小时在线、有独立进程，不是现在的"请求-响应"模式
  • 这两项能力成熟后，第一、二层的补偿方案将自然退出
现在最实用的三个行动建议
01
可视化任务改用 HTML 输出
浏览器自动处理换行和溢出，布局效果稳定可靠，不再需要反复调整 SVG 坐标
02
长周期任务加检查点文件
每个任务执行前先读上次状态，结束后写入进度，从根本上解决执行遗漏问题
03
关注截图回传能力接入
一旦 OpenClaw 接入视觉验证闭环，可视化质量将大幅提升，无需人工干预
